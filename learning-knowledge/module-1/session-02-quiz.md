## БЛОК 1 — Фундамент (аналогия с домом)

**В1.** Вы просите ИИ создать простой интернет-магазин. Всё работает. Через несколько недель вы добавляете систему оплаты, потом личный кабинет, потом онлайн-чат. В один день всё начинает тормозить и постоянно падать. Какова самая вероятная причина?

- А) ИИ плохо написал код
- Б) Основа сайта не была рассчитана на такое количество функций
- В) Просто нужен более мощный компьютер
- Г) На сайт заходит слишком много посетителей

<details>
<summary>Ответ</summary>

**Б)** Это именно аналогия с домом: если фундамент не был рассчитан на 10 этажей, наращивание этажей в итоге создаёт проблемы конструкции. Нужно думать о фундаменте ДО того, как начинать надстраивать.

</details>

---

**В2.** В аналогии с домом, что означает "водопровод"?

- А) Дизайн сайта (цвета, шрифты)
- Б) Движение данных через систему
- В) Скорость загрузки страниц
- Г) Максимальное количество пользователей

<details>
<summary>Ответ</summary>

**Б)** Водопровод — это циркуляция воды в доме. В цифровом мире — это движение данных через систему. Вам не нужно уметь прокладывать трубы — но нужно знать, что они необходимы.

</details>

---

**В3.** Вы создаёте приложение для ведения медицинских карт пациентов. Ваш друг создаёт личный блог. С точки зрения безопасности, какое утверждение правильное?

- А) Обоим нужен одинаковый уровень безопасности
- Б) Блогу нужно больше безопасности, потому что он публичный
- В) Медицинское приложение должно быть гораздо более защищённым, потому что работает с конфиденциальными данными
- Г) Безопасность не важна на старте, её можно добавить потом

<details>
<summary>Ответ</summary>

**В)** Это аналогия с районом: строить в опасном районе (конфиденциальные данные) = бронированные двери, решётки, сигнализация. Строить в спокойном районе (личный блог) = больше свободы и открытости. Проект адаптируется к контексту.

</details>

---

**В4. Верно или Неверно** — "Чтобы хорошо управлять цифровым проектом, нужно уметь самому писать код для каждой части системы."

<details>
<summary>Ответ</summary>

**Неверно.** Не нужно уметь прокладывать трубы. Нужно знать, что они необходимы. Вы должны понимать общую картину, чтобы правильно делегировать, а не делать всё самостоятельно. Без этого общего понимания вы создадите что-то навороченное, но бесполезное и даже опасное.

</details>

<summary>Ответ</summary>

**В6.** Когда вы ставите лайк в Instagram, что происходит в "ресторане"?

- А) Клиент записывает лайк прямо в холодильник
- Б) Клиент говорит официанту (API) "хочу лайкнуть этот пост", API передаёт на кухню, повар сохраняет лайк в холодильнике и подтверждает
- В) Клиент идёт прямо на кухню оставить лайк
- Г) Лайк остаётся только на вашем телефоне

<details>
<summary>Ответ</summary>

**Б)** Клиент (приложение) отправляет запрос через API, сервер (кухня) обрабатывает запрос, сохраняет лайк в базе данных (холодильнике) и отправляет подтверждение клиенту. Клиент НИКОГДА не взаимодействует с холодильником напрямую.

</details>

---

**В7.** Посетитель ресторана просит официанта потанцевать. Официант отказывает. Что это означает в цифровом мире?

- А) Баг в системе
- Б) Пользователь делает запрос, не предусмотренный API — API отказывает, потому что это не входит в его правила
- В) Сервер (компьютер) сломался
- Г) Пользователь не заплатил

<details>
<summary>Ответ</summary>

**Б)** API определяет ограниченный набор возможностей. Всё, что выходит за рамки этих возможностей, отклоняется. "Извините, это не входит в мои обязанности."

</details>

---

**В8.** Кто-то пытается изменить свою почту в Instagram на вашу, чтобы читать ваши сообщения. Что происходит?

- А) API блокирует запрос и не передаёт на сервер
- Б) API передаёт запрос, сервер (кухня) проверяет и отказывает, потому что у пользователя нет прав
- В) Instagram не обнаруживает такую манипуляцию
- Г) API принимает, потому что запрос правильно сформулирован

<details>
<summary>Ответ</summary>

**Б)** API делает свою работу: передаёт. Кухня (сервер) делает свою: проверяет права доступа и защищает. Сервер отвечает: "У вас нет прав на доступ к этим сообщениям."

</details>

- В) Приложение остаётся на серверах Apple

<details>
<summary>Ответ</summary>

- А) Бот быстрее, чем сайт
- Б) Бот более защищён, чем приложение
- В) Не нужно создавать клиентский код — платформа (Telegram/Discord) уже является клиентом, установленным на телефоне пользователя
- Г) Бот может делать больше, чем сайт

<details>
<summary>Ответ</summary>

**В)** Это главное преимущество: у большинства людей уже установлен Telegram или Discord. Достаточно создать кухню (сервер) и дать ссылку. Пользователь взаимодействует напрямую, ничего не скачивая, не заходя на сайт.

</details>

## БЛОК 4 — Кухня и холодильник (Сервер + База данных)

**В12.** Что такое сервер (в IT) конкретно?

- А) Человек, который обслуживает вас в ресторане
- Б) Компьютер, который всегда включён, на котором работает программа, отвечающая на запросы клиентов
- В) Кабель, соединяющий ваш компьютер с интернетом
- Г) Программа, которая устанавливается на ваш телефон

<details>
<summary>Ответ</summary>

**Б)** Сервер — это компьютер, который никогда не выключают. Он всегда включён, на нём работает программа, созданная для обработки запросов клиентов. Он ждёт, и как только кто-то совершает действие (заходит на сайт, нажимает кнопку), он получает запрос и знает, как ответить.

</details>

---

**В13.** В чём разница между поваром и холодильником в цифровом мире?

- А) Разницы нет, это одно и то же
- Б) Холодильник хранит сырые данные, повар (программа) превращает их в полезный результат для клиента
- В) Повар хранит данные, а холодильник их обрабатывает
- Г) К холодильнику может обращаться клиент, к повару — нет

<details>
<summary>Ответ</summary>

**Б)** Холодильник (база данных) = сырые ингредиенты (имена, почта, фотографии, лайки). Повар (серверная программа) = берёт эти ингредиенты, комбинирует, обрабатывает и делает из них блюдо (результат), отправляемое клиенту. Как повар, который берёт курицу, специи и масло, чтобы приготовить готовое блюдо.

</details>

---

**В14. Верно или Неверно** — "Пользователь (клиент) может напрямую обращаться к базе данных для получения своей информации."

<details>
<summary>Ответ</summary>

**Неверно.** Клиент НИКОГДА не имеет прямого доступа к холодильнику. Он не может зайти на кухню и обслужить себя сам. Он должен обратиться к официанту (API), тот передаёт повару (программе), а повар сам идёт к холодильнику (базе данных) за тем, что нужно.

</details>

---

**В15.** Когда вы открываете Instagram и видите свою ленту, что происходит по порядку?

Расставьте в правильном порядке:

- [ ] Сервер (повар) собирает посты и отправляет готовыми к просмотру
- [ ] Ваше приложение (клиент) показывает ленту
- [ ] Сервер (повар) идёт в базу (холодильник) за постами, подходящими для вас
- [ ] Ваше приложение (клиент) запрашивает ленту через API

<details>
<summary>Ответ</summary>

1. **Ваше приложение (клиент) запрашивает ленту через API**
2. **Сервер (повар) идёт в базу (холодильник) за постами, подходящими для вас**
3. **Сервер (повар) собирает посты и отправляет готовыми к просмотру**
4. **Ваше приложение (клиент) показывает ленту**

Это всегда один и тот же поток: Клиент → API → Сервер → База данных → Сервер → API → Клиент.

</details>

## БЛОК 5 — API

**В16.** API определяет три вещи. Какие?

- А) Цену, скорость и дизайн
- Б) Что клиент может запрашивать, формат обмена данными и права доступа
- В) Количество пользователей, размер базы и цвет сайта
- Г) Язык программирования, операционную систему и браузер

<details>
<summary>Ответ</summary>

**Б)** API определяет:
1. **Что клиент может запрашивать** (и что не может)
2. **Формат обмена** (как сформулировать запрос, как прочитать ответ)
3. **Права доступа** (кто что имеет право делать)

</details>

**В17.** Что было бы без API (без официанта в ресторане)?

- А) Всё работало бы так же, только быстрее
- Б) Клиент общался бы напрямую с кухней — был бы полный хаос
- В) Клиенту больше не нужна была бы кухня
- Г) Кухня работала бы лучше без посредника

<details>
<summary>Ответ</summary>

**Б)** Без API клиент общался бы напрямую с кухней — был бы хаос. API структурирует, фильтрует и организует коммуникацию. Именно благодаря ему всё остаётся упорядоченным и защищённым.

</details>

## БЛОК 6 — Сценарии применения

**В18.** Вы создаёте Telegram-бота для ответов на вопросы клиентов. Что здесь является КЛИЕНТОМ?

- А) Ваш код, который работает на сервере
- Б) База данных с вопросами и ответами
- В) Сам Telegram — приложение на телефоне пользователя
- Г) API Telegram

<details>
<summary>Ответ</summary>

**В)** Клиент — это сам Telegram, приложение, которое уже установлено на телефоне вашего пользователя. Именно через него человек видит сообщения и нажимает кнопки. Вам не нужно создавать клиентский код — Telegram уже является клиентом (залом ресторана).

</details>

---

**В19.** В том же Telegram-боте — что является СЕРВЕРОМ (кухней)?

- А) Сам Telegram
- Б) Ваша программа, которая получает вопросы и решает, что ответить
- В) Телефон пользователя
- Г) База данных с FAQ

<details>
<summary>Ответ</summary>

**Б)** Сервер — это ваша программа (код), которая работает на компьютере, всегда включённом. Она получает вопросы через API Telegram, обрабатывает их, ищет ответы в базе данных (холодильнике) и отправляет результат обратно пользователю. Telegram (клиент) только показывает — а ваша программа (кухня) думает и решает.

</details>

---

**В20.** Друг говорит: "Хочу сделать приложение как Uber — просто нужно красивый дизайн." Что не так в его рассуждении?

- А) Дизайн не важен, нужно только хорошо кодить
- Б) Дизайн — это только клиент (зал). За ним нужен сервер (кухня), база данных (холодильник) и API (официант)
- В) Uber вообще не использует серверы
- Г) Достаточно нанять дизайнера и всё заработает

<details>
<summary>Ответ</summary>

**Б)** Дизайн (мобильное приложение) — это только зал ресторана, клиент. Но за ним должны стоять:
- **Кухня (сервер)**, которая управляет логикой: найти водителя, рассчитать маршрут, обработать оплату
- **Холодильник (база данных)**, в котором хранятся аккаунты, GPS-координаты, история поездок
- **API**, которое определяет все взаимодействия между приложением и сервером

И если фундамент не рассчитан на тысячи пользователей, всё рухнет — как дом без фундамента.

</details>
