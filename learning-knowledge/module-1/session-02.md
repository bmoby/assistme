## ВСТУПЛЕНИЕ — Правила игры (10 мин)

### Секрет тех, кто добивается успеха

> "Добиваются успеха те, кто слушает до конца."

Может быть, вы знаете 30%, может быть 50% того, что я скажу. Ловушка: как только вы узнаёте какой-то концепт, ваш мозг отключается. Вы говорите себе "это я знаю" — и пропускаете 70%, которые самые важные.

Результат: вы стартуете на неправильной основе, у вас не хватает кусочков, и вы задаёте вопросы, на которые ответ уже был дан.

**Правило** : даже если вы знаете, даже если вам знакомо то, что я рассказываю — слушайте, запоминайте, оставайтесь сосредоточенными до конца. Обещаю: в тот момент, когда вы возьмёте телефон, чтобы ответить на сообщение или поиграть, я скажу что-то критически важное. А раз вы это пропустили, потом придётся задавать кучу вопросов.

### Вопросы: как это работает

**Пока я говорю :**

- Записывайте свои вопросы по ходу
- Не перебивайте — это ухудшает качество урока, особенно для тех, кто пересматривает видео
- Проявляем уважение и следуем правилам

**Когда я говорю "время для вопросов" :**

- Задавайте всё, что касается текущей темы
- Не выходите за рамки темы — не задавайте вопросов не по теме урока
- Если я отсылаю вас к предыдущему уроку, воспринимайте это позитивно: вы точно знаете, где найти ответ

**Почему это правило?** Если я на сессии 6 отвечаю на вопросы из сессии 1, и если вы на сессии 3 задаёте вопросы из сессии 12, это тормозит всех и ухудшает качество формации для каждого.

**Если вы пропустили уроки** и не понимаете, потому что не хватает базы: приходите задавать эти вопросы мне в личные сообщения.

## ЧАСТЬ 1 — Почему фундамент важен (10 мин)

### Аналогия с домом

Каждый урок важен, но фундамент — важнее всего. Что бы вы ни делали, что бы вы ни разрабатывали, держите эту картину в голове. Именно этим механизмом вы будете думать обо **всех** программах, сайтах, ботах, приложениях.

Без этого вы, конечно, сможете создавать продукты — но эти продукты будут **опасными и плохо спроектированными**.

> "Если я скажу, что у вас могут быть проблемы, вы забудете мои слова.
> А вот если я скажу КАКИЕ ИМЕННО проблемы — тогда вы запомните."

---

**Представьте машину, которая строит любой дом.** Вы выбираете место и говорите: "Построй мне шикарный дом в 3 этажа." Она строит, всё великолепно.

Позже вы просите добавить этаж. Потом ещё один. И ещё. Пока однажды не начинаются:

- **Проблемы с водопроводом** — фундамент не был рассчитан на такую высоту
- **Проблемы с безопасностью** — малейшее землетрясение может обрушить весь дом
- **Каскадные проблемы** — добавляя этажи, вы не подумали о том, что нужно переделать всё остальное

---

### Что нужно запомнить

Нельзя просто наращивать этажи, не думая о том, что было заложено изначально. Вы должны уметь думать о:

- **Водопроводе** — даже не умея устанавливать трубы, вы понимаете, что вода должна циркулировать и подниматься наверх
- **Конструктивной безопасности** — выдержит ли этот дом землетрясение? Это нужно проверять
- **Делегировании** — у вас есть эксперт по безопасности, другой по водопроводу и т.д.

И есть ещё один аспект, о котором нужно думать: **окружение**.

Если вы строите дом в опасном районе, в городе с высокой преступностью — нужно предусмотреть прочные решётки, бронированные двери, систему сигнализации. Вы проектируете дом **с расчётом на ограбление**. Всё закрыто, защищено, заперто.

А если вы строите тот же дом в безопасном месте, в спокойном районе — можно позволить себе больше свободы, больше открытости, панорамные окна, комфорт. Вам не нужна та же степень защиты.

**В цифровом мире точно так же.** Если вы создаёте сервис, работающий с деньгами, медицинскими данными или конфиденциальной информацией — нужно максимально усилить безопасность. А если создаёте простой личный сайт-визитку, вам не нужен такой же уровень защиты. Нужно уметь **адаптировать проект к контексту**.

**Но** : если вы управляете этими экспертами, не понимая картину в целом, вы создадите что-то навороченное, но совершенно бесполезное и даже опасное.

**Ключевая фраза** : "Не нужно уметь прокладывать трубы. Нужно знать, что они необходимы. И нужно знать, в каком районе вы строите."

## ЧАСТЬ 2 — Ресторан: как работает всё цифровое (30 мин)

### Аналогия

Вся IT-сфера, весь цифровой мир работает как ресторан. Все сайты и платформы, которые вы знаете.

У ресторана есть основные составляющие:

```
                                                    +---------------------+
                                                    |   ХОЛОДИЛЬНИК       |
                                                    |   (База данных)     |
                                                    +----------^----------+
                                                               |
+------------------+        +------------------+        +------v-----------+
|                  |        |                  |        |                  |
|   ЗАЛ           | <----> |   ОФИЦИАНТ       | <----> |   КУХНЯ          |
|   (Клиент)      |        |   (API)          |        |   (Сервер)       |
|                  |        |                  |        |                  |
+------------------+        +------------------+        +------------------+
  То, что видит               Правила общения            Повар +
  посетитель,                 между залом                 холодильник:
  красиво и удобно            и кухней                    где всё
                                                          происходит
```

---

### Зал — Клиент

Когда вы входите в ресторан, есть место для приёма гостей. Делается всё, чтобы это место было **красивым и уютным** — чтобы клиент чувствовал себя хорошо.

В IT-сервисах точно так же:

- Instagram
- Любой сайт
- Любая программа
- Автомат для покупки билетов
- Любой сервис, который вы используете на экране

**Клиент — это часть, где показывается только то, что нужно показать, подготовленная для пользователя.**

---

### Кухня — Сервер

На кухне готовят все блюда. Принимают заказы, готовят, отправляют. Всё, что связано с **бизнесом**, всё, что заставляет ресторан **работать**, происходит здесь.

Когда вы нажимаете на кнопку, когда регистрируетесь где-то, когда ставите лайк в Instagram — на этой кухне происходит **очень многое**, чтобы обработать одно ваше действие.

> Всё, что происходит на этой кухне, в цифровом мире — это **сервер**: компьютер, который работает где-то и знает всю бизнес-логику.

---

### Официант (в ресторане) — API

Между клиентом и кухней нужна **коммуникация**. Не повара идут к посетителям — а официанты.

Эти официанты:

- Уже знают, что можно заказать
- Умеют отвечать на вопросы ("У вас есть это? Можно добавить соли?")
- Записывают и передают на кухню
- Предлагают **ограниченный** набор возможностей: сделать заказ, задать вопрос о ресторане — и всё

> Если посетитель попросит официанта потанцевать, тот ответит: "Извините, это не входит в мои обязанности."

**В цифровом мире** совокупность этих правил — что клиент может делать и какая информация может передаваться между клиентом и кухней — называется **API**.

---

### Безопасность: конкретный пример

Можно научить официанта, что если он видит подозрительного человека, нужно сообщить об этом. Он принимает заказ И сообщает о проблеме на кухню. А кухня решает: нужно ли отправить охрану, чтобы вывести этого человека из ресторана?

**Пример с Instagram** : если я как клиент хочу взломать Instagram — например, поменять свою почту на вашу, чтобы читать ваши сообщения — API принимает мой запрос, передаёт на сервер (кухню), и сервер отвечает:

> "Нет, это невозможно. У вас нет прав на доступ к этим сообщениям."

API сделал свою работу: передал. Кухня сделала свою: защитила.

## ЧАСТЬ 3 — Клиент подробно (25 мин)

### Разные названия — один и тот же концепт

В зависимости от того, что мы используем, клиент называется по-разному:

| Контекст | Клиент называется | Описание |
|----------|-------------------|----------|
| **Сайт** (браузер) | **Фронтенд** | Код, отображаемый в вашем браузере |
| **Мобильное приложение** | **Приложение (App)** | Код, скачанный на ваш телефон (App Store, Play Store) |
| **Настольное приложение** | **Программа** | Код, скачанный и установленный на ваш компьютер |
| **Бот Telegram / Discord** | **Сам Telegram / Discord** | Не нужно создавать клиентский код — платформа И ЕСТЬ клиент |

---

### Сайт: фронтенд

Когда мы говорим о сайте, доступном через браузер, клиент — это **фронтенд**. Это часть, которая была написана для пользователя.

### Мобильное приложение

Когда вы заходите в App Store и нажимаете "Загрузить", вы **скачиваете код** на свой телефон. Это и есть ваше приложение. С этого момента оно находится прямо на вашем телефоне. Нажимаете — открывается. Это клиент.

### Настольное приложение

Точно такая же логика на компьютере: вы скачиваете код и запускаете его на своей машине. Даже если отключить интернет, оно не исчезнет — оно на вашем компьютере.

### Боты Telegram / Discord

**Главное преимущество ботов** : не нужно создавать код для взаимодействия с клиентом. Telegram и Discord **уже являются** клиентом.

У большинства людей Telegram или Discord уже установлен на телефоне и компьютере. Достаточно:

1. Создать **кухню** (сервер за ботом)
2. Дать адрес, ссылку
3. Человек нажимает и взаимодействует **напрямую** — без скачивания чего-либо, без перехода на сайт, без изучения чего-то нового

---

### Клиент повсюду

С этого момента, когда вы видите:

- **Банкомат** → это клиент, подключённый к кухне
- **Паспортный контроль** в аэропорту → это клиент
- **Терминал заказа** в Макдоналдсе → это клиент
- **Ваше банковское приложение** → это клиент

Это не обязательно мобильное приложение, не обязательно бот Telegram. Но это всегда часть, созданная для **взаимодействия с пользователем**.

## ЧАСТЬ 4 — Кухня подробно (20 мин)

### Компьютер, который никогда не выключают

Кухня — это попросту **компьютер, который никогда не выключают**. Его часто называют **сервером** — потому что он всегда здесь, всегда включён.

На этом компьютере работает **программа, которую мы создали**, предназначенная для обработки запросов клиентов.

**Конкретный пример** : я создал специальную программу, которая сейчас работает где-то на моём сервере. Она всегда там, ждёт. Как только кто-то заходит на `www.pilotnero.com` и нажимает кнопку, эта программа получает запрос и точно знает:

- Как ответить
- Какую страницу показать
- Что отправить клиенту

**Эта логика одинакова повсюду** : серверы, на которых размещены разные программы, отвечающие нашей бизнес-логике, нашим потребностям.

---

### Холодильник — база данных

Но на кухне есть не только повар. Есть ещё **холодильник**. И в этом холодильнике хранятся все ингредиенты: мясо, овощи, соусы, сыр...

В цифровом мире этот холодильник называется **базой данных**. Это место, где сервер хранит всю информацию: ваши имена, почтовые адреса, фотографии, лайки, сообщения — все "ингредиенты" сервиса.

**Внимание, важное различие** :

- **Холодильник** = сырые ингредиенты (хранящиеся данные)
- **Повар** = работа по превращению ингредиентов в блюдо (программа, обрабатывающая данные)

Повар берёт из холодильника курицу, специи, масло — и делает из этого готовое блюдо, которое отправляет клиенту. Точно так же сервер берёт сырые данные из базы, комбинирует, обрабатывает и отправляет результат клиенту.

**А кто имеет доступ к холодильнику?** Только повар. Клиент **никогда** не имеет прямого доступа к холодильнику. Он не может войти на кухню, открыть холодильник и обслужить себя сам. Он должен обратиться к официанту (API), тот передаёт повару, а повар сам идёт к холодильнику за тем, что нужно.

> Именно так это работает везде. Когда вы открываете Instagram и видите свою ленту, вы не копаетесь напрямую в базе данных Instagram. Ваше приложение (клиент) запрашивает через API, сервер (повар) идёт в базу (холодильник) за постами, которые вам подходят, собирает их и отправляет вам готовыми к просмотру.

```
КУХНЯ (подробно)
+----------------------------------------------------------+
|                                                          |
|   +------------------+        +---------------------+   |
|   |                  |        |                     |   |
|   |   ПОВАР          | -----> |   ХОЛОДИЛЬНИК       |   |
|   |   (Программа)    | <----- |   (База данных)     |   |
|   |                  |        |                     |   |
|   +------------------+        +---------------------+   |
|     Обрабатывает,               Хранит ингредиенты      |
|     преобразует,                (сырые данные)           |
|     готовит блюдо                                        |
|                                                          |
+----------------------------------------------------------+
         ^
         | Клиент НИКОГДА не может
         | получить доступ к холодильнику напрямую
```

## ЧАСТЬ 5 — API подробно (10 мин)

### Правила коммуникации

API — это то, чему мы научили официанта в ресторане. Это **правила коммуникации** между клиентом и кухней.

```
КЛИЕНТ                     API                        СЕРВЕР
  |                         |                           |
  |--- "Хочу X" ---------->|                           |
  |                         |--- Проверяет правила --->|
  |                         |                           |--- Обрабатывает запрос
  |                         |<--- Ответ ----------------|
  |<--- Результат ----------|                           |
  |                         |                           |
```

API определяет:

- **Что клиент может запросить** (и что не может)
- **Формат обмена** (как сформулировать запрос, как прочитать ответ)
- **Права доступа** (кто что имеет право делать)

> Без API клиент общался бы напрямую с кухней — был бы полный хаос.

## ДЕБРИФ — Что нужно запомнить

### Фундаментальная схема

Любая цифровая система, без исключения, следует этой схеме:

```
                                                    +---------------------+
                                                    |   БАЗА ДАННЫХ       |
                                                    |   (Холодильник)     |
                                                    +----------^----------+
                                                               |
+------------------+        +------------------+        +------v-----------+
|                  |        |                  |        |                  |
|     КЛИЕНТ       | <----> |      API         | <----> |    СЕРВЕР        |
|                  |        |                  |        |                  |
+------------------+        +------------------+        +------------------+
  То, что видит             Правила                     Место, где всё
  пользователь              коммуникации                происходит
```

- **Клиент** = зал ресторана (красиво, удобно, для пользователя)
- **API** = официант (правила, ограниченные возможности, передача)
- **Сервер** = кухня / повар (бизнес-логика, решения, безопасность)
- **База данных** = холодильник (хранилище ингредиентов, доступ только у повара)
