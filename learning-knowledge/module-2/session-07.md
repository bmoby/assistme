# Сессия 7 — Память проекта: Git, GitHub, деплой через Vercel и файлы инструкций для ИИ

> Модуль 2, Неделя 4 | Видео перед сессией рекомендуется
> **Артефакт** : проект под Git, загруженный на GitHub, с рабочим URL на Vercel + черновик `AGENTS.md` / `CLAUDE.md` / `GEMINI.md`

---

## Философия С7

**С7 дает проекту память.**

До этого момента студенты уже увидели :

1. что современный проект живет не как один `index.html`, а как настоящее приложение
2. что проект можно запускать локально через `localhost`
3. что в Next.js есть страницы, компоненты, навигация и нормальные ошибки
4. что приложение можно даже вывести в интернет

Но на этом этапе у студентов обычно остается три большие слепые зоны :

- они не понимают, как проект **помнит свое прошлое**
- они не понимают, как проект **становится онлайн не по магии, а по механизму**
- они не понимают, почему ИИ то слушается, то как будто снова работает с нуля

Именно это С7 должна исправить.

### Центральная идея С7

> Как только проект начинает жить во времени, ему нужна память.

У проекта в этой сессии появляется сразу **три слоя памяти** :

1. **Git** — память изменений проекта
2. **GitHub + Vercel** — память и публикация проекта в онлайне
3. **Файлы инструкций для ИИ** — память контекста и правил работы

Это очень сильная логика для новичка, потому что три темы сессии перестают быть разрозненными.

Они собираются в одну картину :

- проект меняется -> нужен исторический след
- проект нужно показать другим -> нужен понятный путь в интернет
- проект ведется с помощью ИИ -> нужен устойчивый контекст для ИИ

### Что важно сказать в самом начале

Нельзя начинать С7 с терминов вроде :

- version control
- repository
- preview deployment
- project rules

Нужно начать с хаоса, который студент уже может почувствовать :

- `final-v2`
- `final-v2-real`
- "что именно я поменял вчера ?"
- "почему у меня на Vercel не то, что у меня локально ?"
- "почему ИИ опять предлагает странное ?"

То есть сначала **проблема**, потом **механизм**, и только потом **слово**.

### Чего нельзя допустить педагогически

После С7 студент не должен уйти с такими ложными выводами :

- "GitHub = Git"
- "commit выкладывает сайт в интернет"
- "если код в GitHub, значит он уже в проде"
- "файл `AGENTS.md` делает ИИ магически умнее"

Вместо этого должна появиться новая карта в голове :

- Git хранит историю
- GitHub хранит эту историю онлайн
- Vercel берет состояние проекта из Git и строит онлайн-версию
- файлы инструкций дают ИИ устойчивый контекст, но не заменяют точный промпт

### Скрытая цель С7

Сделать так, чтобы студент перестал видеть свой проект как хрупкую папку на компьютере.

После этой сессии проект должен начать восприниматься как :

- живая система
- с историей
- с удаленной копией
- с понятным путем в онлайн
- с правилами работы для человека и ИИ

## ВСТУПЛЕНИЕ — Почему проекту нужна память (10 мин)

### 1.1 — Начать с знакомого хаоса

Нужно нарисовать сцену, которая понятна всем :

```text
mon-projet/
  final/
  final-v2/
  final-v2-real/
  final-v2-real-final/
```

И сказать :

> "Когда у проекта нет нормальной памяти, люди начинают имитировать память именами папок."

Это очень важная фраза.

Потому что многие новички уже так делали :

- копировали папку
- меняли название
- боялись сломать рабочую версию
- теряли понимание, что где находится

Именно здесь нужно родить потребность :

> проект должен помнить свои состояния нормально, а не через хаос папок

### 1.2 — Три вида памяти, которые нужны проекту

Дать очень простую структуру :

**Первая память** :

- что менялось в проекте
- когда
- в каком порядке

Это дает Git.

**Вторая память** :

- какая версия проекта сейчас лежит онлайн
- какую версию видят другие

Это строится через GitHub + Vercel.

**Третья память** :

- какому проекту помогает ИИ
- какие команды тут запускать
- какие правила соблюдать

Это дают файлы инструкций.

### 1.3 — Ключевая фраза для лайва

> "Сегодня мы не изучаем три случайные темы. Сегодня мы создаем память проекта."

### 1.4 — Связка с С5 и С6

Можно сказать так :

> "На С5 вы увидели, что проект можно запускать и даже выкладывать. На С6 вы руками начали делать настоящий мини-проект. Теперь этому проекту нужна история, удаленная жизнь и нормальный контекст для ИИ."

## ЧАСТЬ 1 — Git и GitHub без мистики (22 мин)

### 2.1 — Проблема до определения

Нужно поставить очень простые вопросы :

- как вернуться к версии, которая работала ?
- как увидеть, что именно изменилось ?
- как работать спокойно, не копируя весь проект в новую папку ?
- как хранить историю изменений не в голове, а в системе ?

Только после этого давать определение.

### 2.2 — Самое простое определение Git

Лучшее определение для этой сессии :

> "Git — это система, которая хранит историю проекта."

Можно сразу сделать ее еще конкретнее :

> "Git позволяет фиксировать состояния проекта, видеть изменения и при необходимости возвращаться назад."

Очень важно подчеркнуть :

- Git не равен "облаку"
- Git не равен "сайту"
- Git не равен "папке на GitHub"

Git — это именно **система истории**.

### 2.3 — Самая полезная ментальная картинка

Для новичка лучше всего работает такая идея :

> "Каждый commit — это техническая фотография проекта в определенный момент."

Не визуальная фотография.
А фотография состояния файлов.

Можно сказать :

- вы закончили блок работы
- зафиксировали состояние
- подписали его сообщением

И дальше у вас не хаос, а понятная лента :

- здесь добавил навигацию
- здесь создал страницу контакта
- здесь исправил ошибку

### 2.4 — Что такое commit

Для С7 достаточно одного точного определения :

> "Commit — это сохраненное состояние проекта с сообщением."

Нужно сразу разрушить одно заблуждение :

> "Commit не отправляет проект в интернет."

Commit — локальный.
Это нужно повторить несколько раз.

### 2.5 — Git и GitHub — не одно и то же

Это главная путаница сессии.

Формула, которую нужно буквально вбить :

> "Git — это система. GitHub — это платформа онлайн, которая работает с Git."

Очень простая версия :

- Git хранит историю
- GitHub хранит эту историю в интернете и помогает делиться ею

Можно еще сказать так :

- Git = двигатель
- GitHub = место в онлайне, где этот двигатель синхронизируется и показывается другим

### 2.6 — Что студент должен удержать в голове

Вот правильная цепочка :

1. проект существует локально
2. Git хранит его историю локально
3. GitHub хранит удаленную копию этой истории

То есть :

- можно использовать Git без GitHub
- GitHub построен поверх Git

### 2.7 — Минимальный словарь С7

Не нужно вводить слишком много.

Минимум :

- `repo` / `repository` = проект под Git
- `commit` = зафиксированное состояние
- `history` = история commit'ов
- `push` = отправить commits на удаленный репозиторий
- `pull` = получить изменения с удаленного репозитория

Если время ограничено, можно держаться вообще только за три слова :

- Git
- commit
- GitHub

### 2.8 — Главные формулы

Нужно повторять :

- "Git хранит историю проекта."
- "Commit локальный."
- "Push отправляет в GitHub."
- "GitHub — не Git, а платформа вокруг Git."

## ЧАСТЬ 2 — Мини-демо Git : что реально происходит (10 мин)

### 3.1 — Главная цель демо

Цель не в том, чтобы научить студентов всему Git.

Цель гораздо уже :

> показать, что у проекта реально появляется история

### 3.2 — Что показать на экране

Достаточно одной спокойной последовательности :

1. изменить один текст в проекте
2. показать `git status`
3. сделать `git add`
4. сделать `git commit`
5. показать `git log`

Этого достаточно, чтобы родилась правильная картинка.

### 3.3 — Что нужно проговорить вслух

Когда вы показываете `git status`, нужно сказать :

> "Git видит, что проект изменился."

Когда вы показываете `git add`, нужно сказать :

> "Я подготавливаю то, что хочу зафиксировать."

Когда вы показываете `git commit`, нужно сказать :

> "Я сохраняю новое состояние проекта в его историю."

Когда вы показываете `git log`, нужно сказать :

> "Вот теперь история у проекта не в голове, а в системе."

### 3.4 — Минимальная последовательность команд

Можно показать так :

```bash
git status
git add .
git commit -m "Add contact section copy"
git log --oneline
```

Не нужно здесь спорить про :

- `git add .` vs выборочные файлы
- идеальные commit messages
- rebase
- merge conflicts

Для этой сессии это лишнее.

### 3.5 — Самая важная мысль после демо

> "Git дает вам не просто текущую папку проекта. Git дает вам его прошлое."

## ЧАСТЬ 3 — Как деплой через Git и Vercel работает на самом деле (20 мин)

### 4.1 — Какой миф нужно убрать

После первого знакомства с Vercel студенты могут думать :

- "я просто нажал Deploy"
- "GitHub и есть сайт"
- "если код попал на GitHub, значит он уже в проде"

Именно это нужно разложить.

### 4.2 — Самая полезная формула

> "Git управляет версиями. Vercel публикует сайт на основе одной из этих версий."

### 4.3 — Полный механизм простым языком

Нужно дать цепочку без технического шума :

1. вы меняете проект локально
2. делаете commit
3. делаете `push` в GitHub
4. Vercel видит новую версию репозитория
5. Vercel запускает build
6. появляется URL

Это можно показать и в виде схемы :

```text
Локальный проект
  -> commit
  -> push
GitHub
  -> новая версия репозитория
Vercel
  -> build
  -> deploy
  -> URL
```

### 4.4 — Что здесь очень важно не перепутать

Нужно отдельно проговорить :

> "GitHub хранит код. Vercel строит и публикует приложение."

Это не одно и то же.

### 4.5 — Preview и production

Это идеальный момент для простого различия.

**Preview** :

- тестовая версия онлайн
- нужна, чтобы увидеть изменения до основной версии

**Production** :

- основная, официальная версия
- то, что видят реальные посетители

Простая формула :

> "Preview — это версия для проверки. Production — версия для реальных пользователей."

### 4.6 — Что сказать про `main`

Очень простое правило :

- часто основной branch называется `main`
- часто именно он связан с production
- но это настройка, а не закон природы

Для новичка этого достаточно.

### 4.7 — Что сказать про environment variables

Не нужно идти глубоко.

Достаточно сказать :

- иногда сайту нужен не только код
- ему еще нужны параметры среды
- например ключи, URL, секреты
- Vercel умеет хранить их отдельно от кода

Главная мысль :

> "Код сам по себе не всегда равен полностью готовому деплою."

## ЧАСТЬ 4 — Демо : от изменения файла до URL (20 мин)

### 5.1 — Главная цель демо

Сделать видимым один и тот же путь :

> изменение -> commit -> push -> Vercel -> онлайн-обновление

Студент должен увидеть, что деплой — это цепочка, а не магический клик.

### 5.2 — Как лучше показать

Лучше всего взять уже подключенный к Vercel проект, чтобы не тратить время на регистрацию и лишние экраны.

Показать :

1. маленькое изменение в тексте
2. `git status`
3. `git add`
4. `git commit`
5. `git push`
6. GitHub с новым commit'ом
7. Vercel, где появился новый deploy
8. обновленный URL

### 5.3 — Что нужно говорить в каждый момент

Когда показываете изменение локально :

> "Пока что это живет только на моем компьютере."

Когда делаете commit :

> "Теперь это состояние записано в историю проекта."

Когда делаете push :

> "Теперь эта версия ушла в удаленный репозиторий."

Когда открываете Vercel :

> "Теперь Vercel берет эту версию и строит из нее онлайн-приложение."

Когда открываете URL :

> "Вот теперь изменение стало видимым в интернете."

### 5.4 — Если времени мало

Если не хватает времени на branch'и и preview :

- достаточно показать production deploy
- а preview объяснить словами и картинкой

Важно не количество экранов.
Важно, чтобы студенты поняли механизм.

### 5.5 — Самая важная мысль после демо

> "Сайт не оказался в интернете просто потому, что вы написали код. Он оказался в интернете потому, что вы прошли цепочку версии, синхронизации и деплоя."

## ЧАСТЬ 5 — Почему ИИ лучше понимает проект, когда есть файлы инструкций (12 мин)

### 6.1 — Проблема, которую нужно поставить

Студенты уже видели, что ИИ иногда помогает отлично, а иногда как будто не понимает проект.

Это часто происходит не потому, что ИИ "плохой".

Это происходит потому, что :

- каждая сессия ограничена контекстом
- проект большой
- правила проекта нигде нормально не записаны
- человек каждый раз пишет разные формулировки

### 6.2 — Самая простая идея

> "Файл инструкций — это память проекта для ИИ."

То есть вместо того, чтобы каждый раз заново объяснять :

- что это за проект
- какие тут команды
- какие правила кода
- что нельзя ломать

мы пишем это один раз в специальном файле.

### 6.3 — Почему это не магия

Очень важно сказать честно :

> "Файл инструкций не делает ИИ умнее. Он делает задачу яснее."

Почему ИИ начинает работать лучше :

- он получает контекст раньше
- получает его стабильнее
- получает его в повторяемой форме
- меньше путается в ожиданиях

### 6.4 — Как называются эти файлы в разных инструментах

Нужно дать простую карту :

- в **Codex** : `AGENTS.md`
- в **Claude Code** : `CLAUDE.md`
- в **Gemini CLI** : `GEMINI.md`
- в **Cursor** : чаще `.cursor/rules`, иногда `AGENTS.md`
- `.cursorrules` существует как старый формат, но это уже legacy-путь

Главная мысль :

> "Название меняется. Принцип остается тем же."

### 6.5 — Что туда реально писать

Только то, что ИИ полезно знать почти в каждой сессии :

- цель проекта
- стек
- команды запуска
- команды тестов
- важные архитектурные границы
- правила кода
- обязательные проверки

Пример минимального содержания :

```md
# Project

## Goal
- Next.js app for ...

## Commands
- `pnpm dev`
- `pnpm test`

## Rules
- Do not use `any`
- Reuse existing components
- Run tests before commit
```

### 6.6 — Чего туда писать не нужно

Не нужно писать :

- расплывчатые банальности
- временные задачи на сегодня
- противоречивые указания
- все подряд из головы

Плохие примеры :

- "пиши красиво"
- "будь умным"
- "делай качественно"

Хорошие примеры :

- "используй `pnpm test:unit` перед commit"
- "не импортируй напрямую из этой папки"
- "ответы должны быть краткими"

### 6.7 — Самая полезная фраза

> "README объясняет проект человеку. Файл инструкций объясняет проект агенту."

## ДЕБРИФ + ДОМАШКА

### 5 идей, которые нужно унести с собой

1. **Git хранит историю проекта**
2. **Commit локальный, а `push` отправляет изменения на GitHub**
3. **GitHub не равен деплою**
4. **Vercel берет версию проекта из Git и делает из нее онлайн-приложение**
5. **Файлы инструкций дают ИИ устойчивый контекст, но не заменяют точный запрос**

### 3-2-1

- **3 вещи, которые сегодня стали понятнее**
- **2 места, где у вас все еще туман**
- **1 действие, которое вы повторите самостоятельно в ближайшие 24 часа**

### Рекомендуемая домашка

После сессии студенту полезно сделать 5 вещей :

1. еще раз локально изменить проект и сделать один осмысленный commit
2. отправить проект на GitHub или проверить, что он уже синхронизирован
3. заново пройти путь до Vercel и увидеть рабочий URL
4. письменно объяснить себе разницу между `commit` и `push`
5. создать первый файл инструкций для своего инструмента :
   - `AGENTS.md`
   - или `CLAUDE.md`
   - или `GEMINI.md`
   - или правило в Cursor

### Что здесь нужно оценивать

Не :

- количество запомненных Git-команд
- скорость
- идеальность терминологии

А :
