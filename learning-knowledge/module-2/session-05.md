## Философия С5

**С5 переводит студента от статичной страницы к настоящему веб-приложению.**

До этого момента студенты уже изучили :

1. **Как мыслить о цифровом мире** — клиент, API, сервер, база данных
2. **Как работать с файлами и кодом** — IDE, папки, HTML
3. **Как работает браузер** — HTML/CSS/JavaScript, DevTools, Network

На этом этапе они уже умеют :

- создавать HTML-, CSS- и JavaScript-файлы
- открывать их локально в браузере
- менять код, сохранять, перезагружать и проверять результат
- понимать разницу между написанным кодом, прочитанным кодом и исполняемым кодом
- понимать, что компьютер делает ровно то, что написано в коде
- использовать ИИ внутри IDE для создания и изменения файлов

Иначе говоря :

- они уже умеют делать **статичный front-end**
- они уже понимают, хотя пока теоретически, роль сервера, API и базы данных
- у них уже есть нормальное представление о жизненном цикле кода

Теперь нужно сделать следующий шаг :

- понять, что HTML-страница, открытая через `file://`, это еще не настоящее приложение
- понять, что такое **локальный сервер**
- понять, почему разработчики используют **Next.js** и **библиотеки**
- увидеть, что современный проект может работать на собственном компьютере, а потом быть выложен в интернет

**Центральная идея** : сервер — это не "магическое место в облаке". Сервер — это программа, которая запущена на компьютере, ждет, отвечает, следит и исполняет код, пока она включена.

**Главный ментальный переход** :

- `file://` = я открываю файл
- `http://localhost:3000` = у меня на компьютере работает программа, которая отдает приложение

**Угол С5 для новичков** : мы не подаем сервер как абстрактную техническую идею. Мы подаем его как рабочий инструмент разработчика. Локальный сервер позволяет запускать инструмент разработки, который помогает работать : замечает изменения, подготавливает файлы, обновляет браузер и делает работу быстрее, чище и удобнее.

**Почему именно сейчас** : после С4 студенты понимают, что происходит в браузере. С5 показывает вторую половину системы : как заставить настоящее приложение работать на компьютере локально, а потом выложить его в интернет.

**Педагогическое правило С5** : сначала проблема, потом определение. Ни один технический термин не вводится без видимой необходимости.

## ВСТУПЛЕНИЕ — Почему разработчик запускает локальный сервер (12 мин)

### Проблема, которую нужно поставить до любых определений

Начать с очень простой сцены :

- у вас есть HTML-страница
- вы ее меняете
- возвращаетесь в браузер
- обновляете страницу
- потом снова меняете
- потом снова обновляете

Сначала это не страшно.

Но как только проект начинает расти :

- страниц становится больше
- файлов становится больше
- папок становится больше
- логики становится больше
- ручных действий становится больше

И все начинает становиться медленным, хаотичным и утомительным.

### Что нужно донести

Разработчик работает не только "с кодом". Он работает еще и с **инструментами, которые крутятся**.

Иначе говоря :

- мы запускаем локальный сервер на своем компьютере
- на этом сервере мы запускаем инструмент разработки
- этот инструмент помогает строить сайт или приложение

### Простое определение

> "Сервер позволяет программе работать непрерывно, пока сервер включен."

Эта программа может :

- отвечать на запросы
- следить за изменениями
- подготавливать файлы
- помогать в разработке

### Прямое применение к С5

В нашем случае :

- мы запускаем локальный сервер
- мы запускаем на нем инструмент вроде Next.js
- этот инструмент задает структуру, правила и полезные команды

### Первый конкретный плюс, который нужно показать

Самое простое, что нужно дать почувствовать сразу :

- с обычным HTML-файлом страницу нужно обновлять вручную
- с сервером разработки программа сама замечает изменение
- сама подготавливает нужное
- сама обновляет браузер в нужный момент

### Простая фраза для лайва

> "Когда вы разрабатываете, вы не хотите просто открывать файл. Вы хотите запустить инструмент, который работает вместе с вами."

### Ключевая фраза

> "Локальный сервер нужен не только для того, чтобы 'отдавать сайт'. Он нужен еще и для того, чтобы запускать инструменты, которые делают разработку быстрее, чище и организованнее."

### Рекомендуемый устный переход

Можно сказать примерно так :

> "Сегодня вы уже умеете делать HTML-, CSS- и JavaScript-страницы. Вы можете открывать их локально, менять, обновлять, продолжать. То есть да : база для front-end у вас уже есть."

> "Но если вы попытаетесь делать так большой проект, с большим количеством страниц, файлов и логики, вы очень быстро увидите, что это становится неуправляемым."

> "Именно поэтому разработчики создали инструменты для разработчиков. Инструменты, которые работают на локальном сервере, следят за кодом, замечают ошибки, обновляют браузер и организуют проект."

## ЧАСТЬ 2 — Локальный сервер (20 мин)

### Начать с демонстрации

Показать на экране две вещи :

1. Страницу, открытую по адресу :

```text
file:///Users/.../index.html
```

2. Приложение, открытое по адресу :

```text
http://localhost:3000
```

Спросить :

- "Что здесь отличается?"
- "Почему одно начинается с `file`, а другое с `http`?"

### Простое объяснение

**`file://`** означает :

- браузер открывает файл прямо с вашего компьютера
- никакой сервис сейчас не крутится
- браузер просто читает этот файл

**`localhost`** означает :

- у вас на компьютере запущена программа
- эта программа ждет запросов
- браузер разговаривает с ней так же, как разговаривал бы с удаленным сервером

### Добавить идею сервера разработки

На этом этапе сказать прямо :

> "Для разработчика локальный сервер нужен еще и для того, чтобы запускать инструмент разработки."

Такой инструмент может :

- следить за кодом
- замечать изменения
- подготавливать обновленные файлы
- перезагружать или переотправлять нужное в браузер

### Конкретный пример, который нужно вбить

С инструментом вроде Next.js :

- вы меняете код
- локальный сервер видит это изменение
- готовит новую версию
- браузер обновляется без того, чтобы вы все делали вручную

**Результат** :

- быстрее
- удобнее
- организованнее
- лучше подходит для настоящего проекта

### Аналогии

#### Аналогия 1 — Документ vs окошко обслуживания

- `file://` = вы открываете документ, лежащий у вас в ящике
- `localhost` = вы подходите к открытому окошку у себя дома, которое принимает запросы и отвечает

#### Аналогия 2 — Выключенная духовка vs открытый ресторан

- один HTML-файл = рецепт на бумаге
- локальный сервер = кухня, которая включена и готова готовить и отдавать

#### Аналогия 3 — Магазин дома vs магазин в городе

- локальный сервер = ваш магазин открыт у вас дома
- удаленный сервер = тот же магазин открыт на другом компьютере где-то еще

**Механизм один и тот же.** Меняется только компьютер, на котором крутится магазин.

### Определение, которое нужно дать

> "Сервер — это программа, которая работает на компьютере и ждет запросов, чтобы на них отвечать."

### Педагогически важная точка

Повторить несколько раз :

- локальный сервер — не "ненастоящий" сервер
- это настоящий сервер
- логика в локальном и удаленном случае одна и та же
- и во время разработки этот сервер может еще и запускать инструмент, который помогает разработчику работать

## ЧАСТЬ 3 — Библиотеки и Next.js (8 мин)

### 3.1 Почему вообще существуют библиотеки

Начать с простого вопроса :

> "Если разработчики постоянно делают одни и те же вещи, будут ли они каждый раз переписывать все заново?"

Ответ :

- нет
- они переиспользуют уже созданный код
- такой переиспользуемый код мы здесь будем называть **библиотекой**

### Простое определение

> "Библиотека — это полезный код, созданный другими разработчиками, который можно взять и использовать вместо того, чтобы писать все заново самому."

### Конкретный пример

- если вам нужен один и тот же красивый кнопочный стиль
- или уже готовые куски интерфейса
- или уже готовые инструменты

намного разумнее взять готовое, чем каждый раз начинать с нуля.

### Простая формула

> "Библиотека — это уже собранный ящик с инструментами."

### 3.2 Почему Next.js

Отталкиваться от проблемы :

- нам нужно больше, чем один отдельный HTML-файл
- нам нужна настоящая структура проекта
- нам нужны страницы, крепкая основа, локальный сервер и нормальный деплой

Ответ :

- Next.js — это современный инструмент для создания настоящих веб-приложений
- для нашей формации важно прежде всего то, что он дает :
  - понятную структуру
  - организованный проект
  - более удобные обновления во время работы
  - более удобное управление страницами
  - простой деплой
  - возможность позже идти дальше

### Формула для новичка

> "Next.js — это готовый инструмент для создания настоящего веб-приложения."

### Переход к демо Next.js

> "Теперь, когда мы поняли, зачем разработчики используют специальные инструменты, мы будем использовать современный инструмент разработки : Next.js."

## ЧАСТЬ 4 — Пошаговая демо: первый проект Next.js (20 мин)

### Цель

Дать почувствовать первый настоящий момент : "оно реально крутится".

### Демо на экране

Создать проект вместе, например с помощью ИИ, а потом открыть папку проекта.

После этого запустить команду разработки, которая уже предусмотрена внутри проекта :

```bash

```

В этот момент **не объяснять**, что означает `npm`.
Сказать только :

> "Сегодня вам не нужно понимать это слово. Просто запомните : это команда, которая включает сервер разработки для этого проекта."

Потом показать в браузере :

```text
http://localhost:3000
```

### Что нужно сказать ровно в этот момент

> "Сейчас вы не открываете файл. Сейчас вы разговариваете с локальным сервером, который работает на вашей машине."

### Ожидаемая реакция студентов

Они часто будут думать :

- "это просто сайт"
- "это похоже на то, что было раньше"

Тут надо сразу подчеркнуть разницу :

- раньше мы открывали файл
- теперь мы запускаем проект
- раньше локального сервера не было
- теперь работает программа, которая слушает и отвечает

### Микро-изменение вживую

Изменить содержимое главной страницы и сохранить.

Показать :

- браузер обновился
- проект продолжает работать
- сервер разработки сам заметил изменение

### Что важно проговорить вслух

> "В этом и есть один из главных плюсов локального сервера с таким инструментом, как Next.js. Программа следит за кодом, пока вы работаете, готовит обновление и обновляет браузер в нужный момент."

### Что это меняет для них

- меньше ручных действий
- меньше трения
- больше скорости
- лучше организация, когда проект растет

### Ключевая фраза

> "Вы больше не просто смотрите на код. Вы запускаете приложение."

## ЧАСТЬ 5 — Читаем структуру проекта (20 мин)

### Какой страх нужно снять

Когда студенты впервые видят проект Next.js, они часто думают :

- "Это огромно"
- "Тут слишком много файлов"
- "Я сейчас все сломаю"

Цель этой части — упростить картину.

### Показываем только главное

```text
pilote-hub/
  app/
  public/
  ...
```

### Объяснить каждый элемент простыми словами

#### `app/`

- здесь живут страницы приложения
- именно здесь мы больше всего работаем в начале

#### `public/`

- здесь лежат картинки и статические файлы

#### `...`

- есть и другие технические файлы, которые Next.js создает автоматически
- пока нам не нужно понимать их все
- достаточно помнить, что они тоже часть организации проекта

### Аналогия с рестораном

- `app/` = видимая часть ресторана и часть внутренней организации
- `public/` = склад видимых вещей
- `...` = вся техническая часть, уже подготовленная, чтобы ресторан хорошо работал

### Чего делать не нужно

Не надо объяснять весь Next.js.

Цель не в том, чтобы :

- разобрать весь React
- объяснять SSR, продвинутый роутинг или build pipeline

Цель в том, чтобы :

- понять, на что мы смотрим
- перестать паниковать при виде настоящего проекта

### Что стоит спокойно сделать дома

Это не домашнее задание на сдачу.

После сессии студентам нужно просто повторить основные действия у себя на компьютере :

- еще раз открыть свой проект
- несколько раз самостоятельно включить локальный сервер
- несколько раз самостоятельно остановить локальный сервер
- посмотреть, как проект ведет себя после изменений
- спокойно изучить структуру папок Next.js
- попробовать понять, за что примерно отвечают `app`, `public` и другие файлы проекта

### Что важно

- ничего сдавать не нужно
- дедлайна нет
- цель не в том, чтобы все идеально понять сразу
- цель в том, чтобы привыкнуть к новому типу проекта
- главное сейчас — начать уверенно включать, выключать и исследовать локальный проект
